# Oracle: Design Specification (v0.3)

**Project:** Oracle — voice-first SME advisory platform
**Companion documents:** `oracle-product-spec.md`, `oracle-technical-spec.md`
**Audience:** Claude Code (build), designers, frontend engineers
**Last updated:** May 2026

---

## 1. Purpose

This document defines *how Oracle looks, feels, moves, and reads*. The product spec defines what it is. The technical spec defines how it's built. This defines the design language: the tokens, components, motion, copy, and the orb in proper detail.

Everything here is opinionated. The whole point of Oracle is that it doesn't look like every other SaaS app. Defaults are dangerous. When in doubt, follow the specifications here rather than reaching for a Shadcn default.

---

## 2. Design philosophy

### 2.1 The three-word brief

**Calm. Considered. Editorial.**

Every design decision should pass these three filters:

- **Calm:** does this add visual noise, or remove it? Default: remove.
- **Considered:** does this feel intentional, or default? If it could have been generated by a UI library, redesign.
- **Editorial:** does this feel like a magazine, or a dashboard? Magazine wins.

### 2.2 The "evening with your advisor" test

Oracle is used by a managing director at 9pm on a Tuesday with a glass of wine, thinking about their business. Every screen should suit that context. Bright, busy, gamified, notification-heavy interfaces fail this test. Quiet, focused, considered interfaces pass it.

### 2.3 Reference points (with intent)

| Reference | What we take | What we don't |
|---|---|---|
| The Economist (print and app) | Editorial typography, restrained colour, considered hierarchy | Density, multi-column complexity |
| Granola | The calm conversation surface, the listening indicator, the relationship between voice and text | Minimal chrome only when it serves |
| Linear | Restraint, the discipline of removing rather than adding | Tech-bro feel, sharp utilitarianism |
| Arc Browser | Confidence to break SaaS conventions, the spatial sense | Maximalism, the gesture-heaviness |
| Things 3 | The way every detail feels considered, the soft surfaces | The to-do-list metaphors |
| Stripe (marketing) | Premium feel through restraint, perfect typography | Corporate polish, the gradients |
| NYT app | How text and data co-exist, the section dividers | News-product mechanics |

### 2.4 What Oracle is not

- Not a dashboard. There are no widgets, no KPI tiles, no traffic-light status indicators.
- Not a CRM. No tables of records demanding action.
- Not a chatbot. No message bubbles, no chat input field as the primary interface.
- Not a Saas template. No purple gradients, no Inter, no card grids.
- Not a notification machine. No badges, no red dots, no nagging.

## 3. Design tokens

All tokens defined as CSS custom properties on `:root`. Tailwind configured to use them. Frontend engineers should never hardcode values that exist as tokens.

### 3.1 Colour

```css
:root {
  /* Surfaces */
  --color-bg:           #FAFAF7;  /* page background, warm off-white */
  --color-surface:      #FFFFFF;  /* cards, elevated content */
  --color-surface-alt:  #F5F3EE;  /* recessed surfaces, code blocks */

  /* Ink */
  --color-ink:          #1A1A1A;  /* primary text */
  --color-ink-2:        #4A4A4A;  /* secondary text */
  --color-ink-3:        #767676;  /* tertiary, captions, metadata */
  --color-ink-muted:    #A8A6A0;  /* disabled, placeholders */

  /* Lines and edges */
  --color-line:         #E8E6E0;  /* hairlines, dividers */
  --color-line-strong:  #D4D1C8;  /* emphasised divides, focused borders */

  /* Brand */
  --color-teal:         #1E4D5A;  /* primary accent */
  --color-teal-2:       #2E6B7A;  /* hover, gradients */
  --color-teal-3:       #4A8A99;  /* soft fills, washed states */
  --color-teal-bg:      #E8F0F2;  /* subtle teal-tinted backgrounds */

  /* Semantic (sparingly) */
  --color-success:      #3D7A5C;  /* used only for "complete" status */
  --color-warning:      #B8814A;  /* used only in risk register */
  --color-danger:       #A84A3D;  /* used only for critical risks */

  /* Agent accents (for orb gradients — see Section 5) */
  --color-george-warm:  #C9874A;  /* amber inner glow */
  --color-margot-gold:  #B89642;
  --color-iain-indigo:  #3A4C6E;
  --color-priya-mint:   #6B9B8E;
}
```

**Usage rules:**

- The page is `--color-bg`. Cards sit on it without borders, using `--color-surface` and a shadow.
- Teal is rationed. The orb gets it. Primary actions get it. Editorial accents (a section number, a callout border) get it. Everything else is ink and line.
- Semantic colours appear only in the risk register and status indicators. Never as decoration.
- Never use Tailwind's default palette (sky-500, indigo-600 etc). Every colour comes from the tokens above.

### 3.2 Typography

```css
:root {
  /* Families */
  --font-display:  'Tiempos Headline', 'GT Sectra', Georgia, serif;
  --font-text:     'Tiempos Text', 'Source Serif Pro', Georgia, serif;
  --font-ui:       'Söhne', 'Inter', system-ui, sans-serif;
  --font-mono:     'Söhne Mono', 'JetBrains Mono', ui-monospace, monospace;

  /* Scale (display) — for headlines and editorial titles */
  --type-display-xl: 56px / 1.05;   /* hero moments, Board Pack cover */
  --type-display-l:  44px / 1.1;    /* section openers */
  --type-display-m:  32px / 1.15;   /* page headings */
  --type-display-s:  24px / 1.2;    /* card headings */

  /* Scale (text) — for editorial body content */
  --type-text-l:     19px / 1.55;   /* primary body, lede paragraphs */
  --type-text-m:     17px / 1.6;    /* default body */
  --type-text-s:     15px / 1.55;   /* secondary body, metadata captions */

  /* Scale (UI) — for chrome, nav, buttons, forms */
  --type-ui-l:       16px / 1.4;
  --type-ui-m:       14px / 1.4;
  --type-ui-s:       12px / 1.3;
  --type-ui-xs:      11px / 1.3;    /* labels, badges, eyebrow text */

  /* Weights */
  --weight-light:    300;
  --weight-regular:  400;
  --weight-medium:   500;
  --weight-semibold: 600;

  /* Letter spacing */
  --tracking-tight:    -0.02em;     /* display sizes */
  --tracking-normal:    0em;
  --tracking-wide:      0.04em;     /* small UI labels, eyebrows */
  --tracking-wider:     0.08em;     /* uppercase eyebrows */
}
```

**Usage rules:**

- **Display font (serif) is used for:** page titles, section headings inside profile pages, Board Pack content, card-of-the-moment headlines, agent quotes.
- **Text font (serif) is used for:** all editorial body content. Risk descriptions, agent notes, Board Pack paragraphs.
- **UI font (sans-serif) is used for:** navigation, buttons, form inputs, table headers, metadata, timestamps, anything functional.
- **Mono font is used for:** numbers in data displays (tabular figures), code references, ID strings.

The serif/sans split is the single most important typographic decision. Content reads serif. Chrome reads sans. This is what creates the editorial feel.

**Tracking:**
- Display: `--tracking-tight` (large serif looks better tightened)
- Body: `--tracking-normal`
- Eyebrow labels and uppercase tags: `--tracking-wider` (e.g. "RISK REGISTER" above a heading)

**Drop caps:**
- Used in the Executive Summary section of the Board Pack
- First letter of the first paragraph, 4 lines tall, display serif, teal colour
- Nowhere else by default

### 3.6 Glass material

Oracle uses a *Liquid Glass* material system inspired by Apple's design language. Glass is not a finish applied to everything — it is a specific material that appears when surfaces float above other content, react to motion, or need to feel alive without competing with editorial content.

**Three glass tiers, used deliberately:**

```css
/* Subtle: lightest glass, near-invisible, used for floating chrome */
--glass-subtle-bg:     rgba(250, 250, 247, 0.72);
--glass-subtle-blur:   blur(20px) saturate(180%);
--glass-subtle-border: 0.5px solid rgba(255, 255, 255, 0.6);

/* Standard: the default glass surface, used for floating cards and panels */
--glass-standard-bg:     rgba(255, 255, 255, 0.65);
--glass-standard-blur:   blur(32px) saturate(180%);
--glass-standard-border: 0.5px solid rgba(255, 255, 255, 0.5);

/* Heavy: thicker glass for overlays and prominent floating elements */
--glass-heavy-bg:     rgba(255, 255, 255, 0.78);
--glass-heavy-blur:   blur(48px) saturate(200%);
--glass-heavy-border: 0.5px solid rgba(255, 255, 255, 0.45);
```

**Glass usage rules (critical):**

Glass is used for:
- The top navigation bar (subtle glass, becomes more opaque on scroll)
- The walkthrough orb container during Board Pack reading (standard glass at bottom of viewport)
- Modal and side-panel surfaces (heavy glass)
- The handoff card during voice sessions (standard glass)
- Notification toasts (heavy glass)
- The intake "Continue" button background area (subtle glass)
- The mobile bottom action bar (subtle glass)
- The transcript backdrop when transcript is enabled (subtle glass)

Glass is NOT used for:
- The main canvas background (it stays warm off-white)
- Profile pages and editorial content (these are solid surfaces, not floating)
- The Board Pack body content (editorial reading must be on solid surface)
- Buttons themselves (only their containers if floating)
- The orb (it has its own material treatment)
- Anything that contains paragraphs of body text (glass + body copy is hard to read)

**The principle:** glass appears on *chrome that floats above content*. Content itself never sits on glass. If you can't read body text comfortably on a surface, it shouldn't be glass.

**Implementation specifics:**

```css
.glass-standard {
  background: var(--glass-standard-bg);
  backdrop-filter: var(--glass-standard-blur);
  -webkit-backdrop-filter: var(--glass-standard-blur);
  border-top: var(--glass-standard-border);  /* Subtle top edge highlight */
  box-shadow: var(--shadow-glass-inner), var(--shadow-m);
}
```

The *inner highlight shadow* and the *0.5px top border* are what make glass look like glass rather than just transparency. They suggest a refractive edge catching light from above.

**Fallback for browsers without backdrop-filter:**

```css
@supports not (backdrop-filter: blur(20px)) {
  .glass-subtle, .glass-standard, .glass-heavy {
    background: rgba(255, 255, 255, 0.95);  /* Near-opaque fallback */
  }
}
```

**Performance considerations:**

- `backdrop-filter` is GPU-expensive. Use sparingly.
- Avoid stacking multiple glass layers (e.g. glass nav over glass content over glass background)
- On mobile, prefer subtle glass over heavy glass to preserve battery and frame rate
- Test on a mid-range Android device; if it stutters, dial back blur values

**Glass and the editorial principle:**

The temptation with Liquid Glass is to apply it everywhere because it looks impressive. Resist this. Oracle's editorial calm depends on most surfaces being solid, quiet, and warm. Glass is a *special material* that appears when something floats. Used sparingly, it suggests premium quality. Used everywhere, it looks like a 2020 glassmorphism Dribbble shot.

The test: a screen with glass should still pass the "9pm with a glass of wine" check. If the glass feels showy rather than considered, remove it.

**Usage rules:**

- Default card: `--shadow-s`
- Elevated card (card of the moment, modal): `--shadow-m`
- Floating (popovers, dropdowns): `--shadow-l`
- Highest layer (full-screen overlays, the active orb): `--shadow-xl`
- Never combine a shadow with a border. Choose one.

### 3.7 Motion

```css
:root {
  --ease-out:      cubic-bezier(0.22, 1, 0.36, 1);     /* default for entrances, opens */
  --ease-in-out:   cubic-bezier(0.65, 0, 0.35, 1);     /* state changes */
  --ease-organic:  cubic-bezier(0.4, 0, 0.2, 1);       /* orb animations, breathing */

  --dur-instant:   100ms;
  --dur-fast:      180ms;
  --dur-medium:    280ms;
  --dur-slow:      480ms;
  --dur-deliberate: 800ms;
  --dur-glacial:   1600ms;  /* orb breathing, ambient motion */
}
```

**Motion principles:**

1. **Ease out, never linear.** Anything moving uses `--ease-out` or `--ease-in-out`. Linear motion feels robotic.
2. **Nothing snaps.** Never use spring physics that overshoot. Bounces feel toylike, wrong for the audience.
3. **Calm over snappy.** Where in doubt, slower. 280ms over 180ms. The product is not Slack.
4. **Motion has meaning.** Things animate to signal state change, draw attention, or convey continuity. Never animation for its own sake.
5. **Respect reduced motion.** Honour `prefers-reduced-motion`. The orb stops breathing. State changes become instant. The product still works.

**Default usage:**

| Event | Duration | Easing |
|---|---|---|
| Hover state on a button | `--dur-fast` | `--ease-out` |
| Modal open | `--dur-medium` | `--ease-out` |
| Page transition (cross-fade) | `--dur-medium` | `--ease-in-out` |
| Profile update animates in (during voice) | `--dur-slow` | `--ease-out` |
| Card of the moment appears | `--dur-slow` | `--ease-out` |
| Orb breathing | `--dur-glacial` | `--ease-organic` |
| Orb state change | `--dur-deliberate` | `--ease-organic` |
| Agent handoff (orb morph) | `--dur-deliberate` | `--ease-organic` |
| Toast/notification slide | `--dur-medium` | `--ease-out` |

## 4. Layout system

### 4.1 The canvas (home screen)

The canvas is the single most important layout in Oracle. It is not a dashboard.

**Desktop layout (top to bottom):**

```
┌───────────────────────────────────────────────────────┐
│  [Oracle wordmark]              [business name] [⚙]  │  ← Top nav, 64px height
├───────────────────────────────────────────────────────┤
│                                                       │
│                                                       │
│                       [orb]                           │  ← Centre column, generous vertical space
│                  (200-280px diameter)                 │
│                                                       │
│                                                       │
│        ┌───────────────────────────────────┐          │
│        │  [Card of the moment]              │          │  ← Single card, 680px max width
│        │  body, action button                │          │
│        └───────────────────────────────────┘          │
│                                                       │
│        [tap to speak] or [start a session]            │  ← Primary CTA, subtle
│                                                       │
│                                                       │
│                  ─── scroll for more ───              │  ← Below the fold indicator
│                                                       │
├───────────────────────────────────────────────────────┤
│                                                       │
│  [Profile section: Strategy]                          │  ← Below the fold, optional engagement
│  ...                                                  │
│                                                       │
└───────────────────────────────────────────────────────┘
```

**Key principles:**

- The orb is roughly 40% from the top, optically centred.
- The card of the moment sits below the orb, never above.
- There is nothing in the visual margins. No sidebars, no widget rails.
- The profile is below the fold. The leader has to choose to engage with it. The canvas itself is calm.
- Total height above the fold: 100vh. The leader sees the orb, the card, the CTA — nothing else — when they first land.

**Top navigation glass treatment:**

The top nav is one of the deliberate places glass appears. On initial load it is a solid surface (`--color-bg`, no separator). When the leader scrolls and content passes beneath it, the nav transitions to subtle glass:

- Below 80px scroll: solid bar, no border, no shadow
- 80px to 120px scroll: cross-fade to glass over the scroll distance
- Above 120px scroll: full subtle glass treatment (`--glass-subtle-bg`, blur 20px), with a 0.5px bottom edge in `--color-line` for definition
- Returning to top: reverse transition

This creates the Apple-like effect of the navigation feeling like a physical surface that reveals content moving underneath it. Used only here on the canvas; on profile pages and the Board Pack the nav is solid by default (editorial reading shouldn't have shifting chrome above it).

**Mobile layout:**

```
┌─────────────────────┐
│ [≡]  [Oracle]   [👤]│  ← Top nav, 56px, glass on scroll
├─────────────────────┤
│                     │
│                     │
│        [orb]        │  ← 160-200px
│                     │
│                     │
│  ┌──────────────┐   │
│  │ Card of the  │   │
│  │ moment       │   │
│  └──────────────┘   │
│                     │
│  ┌─────────────┐    │  ← Bottom action bar: subtle glass
│  │Tap to speak │    │     floats above content
│  └─────────────┘    │
└─────────────────────┘
```

Same principles. The orb and card scale down but their relative positions hold. The mobile bottom action bar is the second deliberate glass placement: a subtle glass surface holding the primary action, floating above content as the leader scrolls.

### 4.4 The intake flow (pre-onboarding)

The intake is the leader's first impression of Oracle's design language. Five questions, one per screen, no chrome.

```
┌───────────────────────────────────────────────────────┐
│  Oracle                                                │  ← Wordmark only, top-left
│                                                       │
│                                                       │
│                                                       │
│                                                       │
│        What does your business do?                    │  ← Display L, serif, centred
│                                                       │
│        ┌─────────────────────────────────────────┐    │
│        │ Type to search...                       │    │  ← Single input, no border, soft underline
│        └─────────────────────────────────────────┘    │
│                                                       │
│        A short sector or two is plenty.               │  ← Optional helper, ink-3, type-ui-s
│                                                       │
│                                                       │
│                                                       │
│                                            Continue → │  ← Bottom-right, tertiary button feel
│                                                       │
└───────────────────────────────────────────────────────┘
```

**Specifications:**

- Background: `--color-bg`
- Centred vertically (the input is at optical 40% from top)
- No progress bar, no "step 3 of 5", no surrounding chrome
- Input: no border, just a thin `--color-line-strong` underline; focuses to `--color-teal`
- "Continue" appears only after input is valid; cross-fades in over 300ms
- Transition between questions: 400ms cross-fade, the previous question fading out before the next fades in
- Mobile: same structure, larger touch targets, input at 50% from top

**Why this matters:** the intake sets expectation. A form that feels like Typeform is acceptable. A form that feels like Stripe Atlas is the target.

### 4.6 Loading states

Three contexts, three treatments.

**Context A: Page load (initial route resolution, app shell)**

- Show the page shell (top nav, layout grid)
- Where content will be: a subtle text "Loading…" in `--type-ui-m`, `--color-ink-3`, centred
- No spinners, no skeleton screens, no shimmer
- If load exceeds 1.2 seconds: the text changes to "Just a moment…"
- If load exceeds 4 seconds: an error state appears (see 4.7)

**Context B: Inline async (a section refreshing, profile updating)**

- The section title and structure remain visible
- Where data should be: a single line, "Loading…" in `--type-ui-s`, `--color-ink-3`, left-aligned where data would start
- The orb is unaffected — it remains in whatever state it's in

**Context C: Long-running generation (Board Pack, session post-processing)**

This is the only context where Oracle shows extended progress. Specifically for Board Pack generation:

```
            [small orb, 80px, in thinking state]


            Drafting your May read…

            This will take a moment.
```

- The orb appears in thinking state, smaller than canvas size
- Centred vertically in the available space
- Text below in display S, serif, `--color-ink-2`
- Sub-text in `--type-text-s`, `--color-ink-3`
- No progress bar, no percentage, no spinner
- After 90 seconds: text changes to "Almost done…"
- After 180 seconds: switches to error state with retry

**Never use:**
- `animate-pulse` on placeholder blocks (Tailwind default)
- Spinning circles
- Three-dot loading animations ("...")
- Progress bars with fake progress
- Generic "Loading data" copy

### 4.7 Error states

The product has three failure modes, each designed.

**Mode A: Connection or system error (recoverable)**

When something fails but Oracle can probably recover:

```
            [orb in error state, desaturated]


            Brief hiccup.

            We've lost connection for a moment.
            Trying to reconnect now.

            [Try again]
```

- Orb desaturates to roughly 30% colour
- A small wifi-off icon (Lucide, `--color-ink-3`) appears below the orb at 16px
- Display S, serif, `--color-ink` for headline
- Body in `--type-text-m`, serif, `--color-ink-2`
- Primary action "Try again" or "Try later" depending on context
- After 30 seconds of retry attempts: escalates to Mode B

**Mode B: Service unavailable (extended)**

When something is genuinely broken and won't recover quickly:

```
            [orb static, fully desaturated]


            We need a moment.

            Something on our side isn't working
            quite right. We've been alerted and
            we'll have it sorted shortly.

            In the meantime, you can still read
            your profile and the latest Board Pack.

            [View profile]    [Contact support]
```

- Orb is static (no breathing, no animation)
- Body explains, offers an alternative path
- Two equal-weight actions: a useful fallback + support contact
- An incident reference number appears in `--type-ui-xs`, `--color-ink-3`, below the actions

**Mode C: User-error or limitation (not Oracle's fault)**

For things like "you've used all your sessions this month" or "this email is already registered":

```
            EYEBROW

            A brief note.

            You've had five sessions this month, which
            is our current limit during the trial.
            George will be back next month — or if you'd
            like more time now, we can extend.

            [Request more sessions]    [Wait until next month]
```

- No orb animation change — this is a content-level message, not a system error
- Eyebrow + display S heading + body, all in standard editorial type
- Always offers a path forward (no dead ends)

**Forbidden across all error states:**
- Red error backgrounds
- Sad emojis or apologetic illustrations
- Stack traces or technical error codes shown to users (logged, never displayed)
- "Oops!" or any informal apology lexicon
- Modal dialogs blocking the entire UI for non-critical errors

## 5. The orb (in proper detail)

The orb is Oracle's visual signature. Every detail matters.

The orb shares material kinship with the Liquid Glass system but is its own thing. Where glass surfaces refract content behind them, the orb is a self-contained luminous sphere. The shared visual language: soft inner highlights catching light, subtle edge definition, the sense of a real material rather than a flat shape.

### 5.1 Anatomy

The orb is built from four layered SVG elements:

1. **Outer halo** — a soft, large radial gradient creating ambient glow. Extends 60-80% beyond the core. Very low opacity (0.15-0.25).
2. **Core sphere** — the main circle. Radial gradient from a brighter centre to a deeper edge, suggesting depth without 3D.
3. **Inner highlight** — a small, offset gradient inside the core, suggesting a light source from upper-left. This is what gives the orb its sense of being a physical, illuminated material rather than a flat coloured disc.
4. **Specular edge** — a very thin (0.5px equivalent), brighter ring around the very top of the core, like the highlight on a glass sphere catching room light. Sits at roughly 12 o'clock, fading off by 4 and 8 o'clock.

Optionally a fifth element during specific states:

5. **Ripple ring** — a thin circle stroke that expands and fades during listening state.

The specular edge is what visually ties the orb to the Liquid Glass material system. It is subtle — most viewers will not consciously notice it — but it is the difference between an orb that looks like a circle with a gradient, and one that looks like an actual luminous sphere.

### 5.2 Size

| Context | Diameter |
|---|---|
| Canvas (desktop, idle) | 240px |
| Canvas (desktop, active) | 280px |
| Canvas (mobile, idle) | 160px |
| Canvas (mobile, active) | 200px |
| Inside a card (e.g. preview) | 48px |
| Top nav indicator (when session active in background) | 24px |

The orb scales slightly when transitioning from idle to active. Adds a sense of "leaning forward" into the conversation.

### 5.4 States

#### Idle

Slow, calm breathing. The orb is alive but unobtrusive.

- Core scale animates between 1.0 and 1.025 over 4.8 seconds (`--dur-glacial` × 3)
- Halo opacity animates between 0.85 and 1.0, slightly offset (half a cycle behind the core)
- Inner highlight stays static
- Easing: `--ease-organic`, looping infinitely

#### Listening (user is speaking)

The orb reacts subtly. Not a microphone visualiser. A sense of presence.

- Core scale animates between 1.02 and 1.04, faster (1.2s cycle)
- A single ripple ring expands every 2 seconds: scale 1 → 1.4, opacity 0.4 → 0
- Halo brightens by 15% (opacity multiplier 1.15)
- Inner highlight unchanged

The ripple is the only "I'm listening" affordance. Never a literal mic icon.

#### Speaking (agent is speaking)

The orb responds to audio amplitude. This is the most reactive state.

- Core scale driven by smoothed audio amplitude: base 1.0, max 1.08
- Smoothing: 100ms moving average to prevent jitter
- Halo opacity also driven by amplitude: base 0.9, max 1.0
- Inner highlight shifts position slightly with amplitude (up to 4px offset)
- Easing on amplitude response: `--ease-organic`

The orb should feel like it's *vocalising*, not VU-metering. Subtle, not theatrical.

#### Thinking (processing, tool call running, brief pause)

Distinct from speaking. The orb shows it's working without claiming to speak.

- Core scale: gentle pulse, 1.0 → 1.015 → 1.0 over 1.4 seconds
- Halo shifts to a slightly cooler tone (reduce saturation by 20%)
- Inner highlight dims slightly
- No ripple

This state should appear during tool calls, between sentences, or when the LLM is generating. Maximum 3-4 seconds of this state before something else happens; if longer, escalate to a "still working..." subtitle below the orb.

#### Handing off (transitioning between agents)

The single most cinematic moment in the product. Handle with care.

- Duration: 800ms total
- 0-300ms: current orb contracts to 0.92 scale, halo dims to 0.3 opacity
- 300-500ms: gradients cross-fade to the new agent's palette (core, highlight, halo all transition)
- 500-800ms: new orb expands to 1.0 scale, halo restores to full

During the transition, a faint subtitle appears below the orb: "Margot is joining…" — fades in 200-500ms, fades out 700-1000ms.

#### Error / disconnected

Rare but must be designed.

- Core desaturates by 70% (becomes nearly grey)
- Halo disappears entirely
- A small icon (Lucide `wifi-off` or similar) appears below the orb in `--color-ink-3`
- Subtitle: "We've lost the connection. Trying to reconnect…"

### 5.6 Profile updates during voice (the peripheral magic)

When an agent calls a tool during voice — `createRisk`, `updateObjective`, `createOpportunity` — the corresponding profile section must update visibly but peripherally. This is the visible magic of voice-first.

**Principle:** the leader is focused on the orb. Profile updates happen in the periphery of their attention. They should *notice* the update without it interrupting the conversation.

**The animation:**

When a new record is created during voice:

1. The relevant section count badge increments with a number-flip animation (300ms, `--ease-out`)
2. A soft teal glow pulses at the edge of the section (or on the section nav item if section isn't visible)
3. If the section is visible: the new record fades in at the top, sliding down 8px to its position over 480ms
4. After 600ms: a small subtitle appears below the orb: "Iain added a risk" — fades in over 200ms, sits for 2s, fades out over 600ms

When an existing record is updated:

1. The record's container briefly elevates (shadow shifts from `--shadow-s` to `--shadow-m`) over 300ms
2. A subtle teal hairline appears on the left edge of the record for 1.5s, then fades
3. No subtitle (it's a smaller event)

When a record is deleted:

1. The record fades to 0.3 opacity over 200ms
2. Collapses height with `--ease-in-out` over 400ms
3. Subtitle below orb: "Closed a risk" or similar

**Specifications:**

- All animations respect `prefers-reduced-motion` (collapse to instant transitions)
- Animations queue, never overlap — if three risks are added in quick succession, they animate sequentially with a 200ms stagger
- On mobile where the section isn't visible: the bottom nav item for that section pulses gently (3 pulses, 800ms each)
- Subtitle below the orb uses `--type-ui-s`, `--color-ink-3`, italic, centred, 32px below the orb

**Why this matters:** in voice-first, the leader can't see what's being captured. This peripheral feedback is how trust is built. Without it, the leader either over-checks (breaking the flow) or worries George isn't capturing things.

### 5.7 The "first contact" animation

When the leader lands on the canvas for the very first time (post-intake, pre-onboarding), the orb does a special introduction:

1. Page loads with the orb at 0 scale, 0 opacity
2. After 200ms: orb fades in and scales to 0.8 over 600ms
3. After another 400ms: orb scales to 1.0 with a soft halo bloom (halo opacity 0 → 1 over 800ms)
4. After another 600ms: George says "Hello [name]…"

This animation runs exactly once per user.

### 5.8 Accessibility for the orb

- All states must work with `prefers-reduced-motion`. In reduced motion: orb is static, state changes happen as discrete colour shifts (no breathing, no scaling, no ripples).
- Orb has an aria-label that updates with state: "Oracle Advisor George, idle" / "George is listening" / "George is speaking" / "Margot is joining".
- The transcript display (Section 6.5) is the accessible equivalent for users who can't see or hear the orb.

### 5.9 The orb in different contexts

The orb appears in places other than the canvas centre. Behaviour varies:

| Context | Size | Animation |
|---|---|---|
| Canvas centre (primary) | 240-280px | All states active |
| Inside a card preview | 48px | Idle only, slower breathing (cycle 2x slower) |
| Top nav (session in background) | 24px | Pulse only when speaking |
| Board Pack walkthrough (fixed bottom) | 80px | All states, smaller amplitudes |
| Loading state (Board Pack generation) | 80px | Thinking only |
| Mobile keyboard active | hidden | N/A |

All variants share the same gradients and colour treatments — only size and animation amplitude scale.

### 6.5 The transcript

During a voice session, an optional transcript display. Designed for accessibility and for moments when the leader wants to see what's being said.

**Default state:** off. The leader sees only the orb.

**When enabled:** sits below the orb, ghostly.

```
                  [orb]


            "...so the supplier concentration is
            real, but it's not new. It's been like
            this for two years."

                                          — Iain
```

**Specifications:**

- Single line of speaker visible at a time, plus the previous speaker's last line above it (faded to 0.5 opacity)
- Typography: `--type-text-l`, serif, `--color-ink-2`, italic for agent speech
- Speaker label: small em-dash + name, right-aligned, `--type-ui-s`, `--color-ink-3`
- Position: centred, 32px below the orb on desktop

**Handling STT reality (this is critical):**

Real-time speech-to-text produces messy output: partial words, mid-sentence revisions, latency, jitter. The transcript design must handle this gracefully.

1. **Partial transcripts (text being refined in real time):** display at 0.6 opacity. As the transcript stabilises, opacity increases to 1.0.
2. **Revision behaviour:** when STT changes earlier words (which it will), the change is smooth — fade old word out (200ms), fade new word in (200ms), no jarring text-replacement.
3. **Latency tolerance:** display catches up to actual audio with a deliberate 500ms delay. This avoids showing premature partial text. The trade-off is the transcript runs slightly behind the voice, which feels more like a film subtitle than a live caption.
4. **Sentence breaks:** when the STT signals end-of-utterance, the current line "commits" (fades from italic to upright, sits for an additional 2 seconds) before the next line begins.
5. **Long pauses:** if no speech for 4+ seconds, transcript clears entirely. The orb's state shows what's happening.

**Forbidden:** chat bubbles, message lists, scrollback in the live view. This is a film subtitle, not a messenger.

**Post-session:** the full transcript is available in the session history, where it *does* render as a structured back-and-forth document (but still editorial, not a chat log).

**Toggle:** a tiny "show transcript" pill below the orb when off, that the leader can tap. Subtle, easy to ignore. State persists per user (if they enable it once, it stays on for future sessions).

**Forced display:** if the leader has accessibility settings indicating they need captions, the transcript is on by default and cannot be hidden.

## 7. The Board Pack (design specification)

Because this is the headline deliverable, it gets its own section.

### 7.1 Structure (top to bottom)

The Board Pack is organised around four pillars — Money, Customers, How You Run Things, and Your People — that quietly mirror the four quadrants of the Balanced Scorecard (Financial, Customer, Internal Process, Learning & Growth). The leader never sees the framework name; they see plain-language pillars. But the structure is rigorous behind the scenes, which is what gives the output its credibility.

The full sequence:

1. **Cover** — full-bleed, calm, sets the tone
2. **Section 01: The Headline** — executive summary (one-page narrative across all four pillars plus risk)
3. **Section 02: Money** — financial health, ROI of recent decisions, key ratios (light touch for PoC where Finance specialist isn't yet built)
4. **Section 03: Customers** — strategic objectives, growth opportunities, customer-facing changes
5. **Section 04: How You Run Things** — operational health, top risks, process improvements
6. **Section 05: Your People** — capability, culture, learning, change readiness
7. **Section 06: What I'd Focus On This Month** — the ROI-backed action plan: prioritised list of 3-5 specific moves, with expected benefit and effort
8. **Section 07: Where We Might Want Help** — Valorem First handoff suggestions (if applicable)
9. **End plate** — "End of report. Reach out when ready."

**Pillar labelling principle:** plain language always wins. "Money" not "Financial Perspective". "Your People" not "Learning & Growth". "How You Run Things" not "Internal Process". The leader recognises their own business in these labels; they would not recognise it in Kaplan and Norton's original vocabulary.

**Insight schema underpinning each pillar:**

Each item discussed in the Board Pack (a risk, an opportunity, a recommendation) is structured behind the scenes using the 11-field insight schema:

- Pain point / observation
- Evidence (specific examples)
- Impact area (Finance, Customer, Process, People)
- Root cause
- Severity (low / medium / high / critical)
- Frequency (one-off / repeated / systemic)
- Financial impact estimate
- Strategic relevance
- Recommended action
- Expected benefit
- Confidence level

The Board Pack body does not show these fields as a list; it weaves them into editorial prose. But the underlying data is always present and queryable, which means recommendations always answer "so what, and why now?"

### 7.8 The walkthrough sync

When the leader chose "Begin walkthrough":

**Default behaviour:**
- The orb appears fixed at the bottom-centre of the viewport, 80px diameter, inside a standard-glass container (a pill-shaped surface roughly 200x100px). The glass holds the orb plus a small subtitle area below it, floating above the Board Pack content
- The glass container has `--glass-standard-bg`, `--glass-standard-blur`, `--radius-xl`, and `--shadow-m` — it should feel like a floating control panel hovering above the editorial content
- The page auto-scrolls smoothly to keep the section George is reading centred in the viewport
- The current section gets a subtle teal accent on its section number (01 in teal instead of ink-3)
- The leader can interrupt at any time by tapping the orb
- Progress is shown via a thin teal bar at the very top of the viewport (1px tall, fills left-to-right as George progresses)

**When the leader interrupts (taps orb mid-narration):**
- George pauses immediately
- Orb transitions from speaking to listening state
- Auto-scroll stops at current position
- A small subtitle appears below the orb: "Go ahead…"
- After the leader's question and George's answer, George asks "Want me to continue from where I was, or pick somewhere else?"

**When the leader scrolls away from George's position (manual scroll):**
- Auto-scroll suspends
- George keeps reading; a small floating indicator appears at the side of the viewport: "George is reading section 03 — tap to return"
- If the leader doesn't return within 30 seconds: George politely asks "Should I keep going, or are you reading ahead?"
- The leader can resync at any time by tapping the indicator

**When the leader scrolls ahead:**
- Same as above; George doesn't skip ahead
- George can offer to skip if asked ("Want me to jump to section 5?")

**When the leader wants George to mute but keep the visual sync:**
- A small "Read silently" affordance appears near the orb during walkthrough
- Tapping it: orb shrinks to top-nav indicator size, narration stops, but the leader can manually scroll and the section accents still highlight where they are
- Orb can be expanded again at any time to resume

**When the walkthrough completes:**
- Final section ("End plate") appears
- George says a closing line ("That's where we are. What would you like to talk about?")
- Orb scales up to normal canvas size and remains active for follow-up conversation
- The Board Pack remains accessible in the leader's history

**Reduced motion:**
- Auto-scroll is disabled; sections highlight as George reaches them but the leader controls scroll position
- Orb does not animate between states (instant transitions)
- The "George is reading section X" indicator becomes more prominent to compensate

## 9. Voice and tone (written copy)

Every word in the product. Notifications, buttons, error states, empty states, the Board Pack body.

### 9.1 The principles

1. **Write like George talks.** Warm, direct, slightly British, considered. Never American-startup-perky.
2. **No platform-speak.** "Submit" → "Save". "Engage" → "Talk to George". "Manage" → "Review".
3. **Use the leader's words, not consultants' words.** "Things you might do" not "strategic initiatives". "Things that could go wrong" not "risk vectors".
4. **Never feature-name in UI copy.** Don't say "Open the Board Pack module". Say "Read your May Board Pack".
5. **Avoid the second-person imperative.** "Add a new risk" feels like an order. "Let's add another" or "What else is on your mind?" feels like a conversation.

### 9.2 Specific copy patterns

**Empty states:**

Instead of: "No risks have been added yet."
Write: "Iain hasn't heard about any risks yet. Want to talk through what's keeping you up at night?"

Instead of: "No objectives."
Write: "Margot and I haven't mapped out your objectives yet. We'll do that in our first proper conversation."

**Buttons:**

Instead of: "Submit", "Save", "Continue", "Cancel"
Write: depending on context:
- "Save this risk" / "Capture this" / "Done for now"
- "Talk to George" / "Start a session" / "Pick this up later"
- "Not now" instead of "Cancel" (less final)

**Notifications:**

Instead of: "Reminder: review supplier risk"
Write: "Iain's been wondering about supplier risk. Free for a 10-minute conversation?"

**Errors:**

Instead of: "Connection failed. Please try again."
Write: "We've lost the connection for a moment. Trying to reconnect…"

Instead of: "Something went wrong."
Write: "Something didn't quite work. George will pick this back up shortly."

**Confirmation:**

Instead of: "Are you sure you want to close this risk?"
Write: "Closing this risk means we'll stop tracking it. Sound right?"

### 9.3 Tone by context

| Context | Tone | Example |
|---|---|---|
| Onboarding | Welcoming, unhurried | "Take your time. We've got nothing on the clock." |
| Routine session | Familiar, easy | "Good to see you. What's on your mind?" |
| Proactive nudge | Gentle, not pushy | "When you have a moment, there's something Iain wanted to mention." |
| Board Pack delivery | Anticipatory, considered | "Your May read is ready. Want me to walk you through it?" |
| Valorem First handoff | Honest, useful | "This is past where I can help on my own. Worth a proper conversation with the VF team." |
| Error | Calm, reassuring | "Brief hiccup. Won't lose anything." |

### 9.4 Per-agent written voice

When an agent writes (in cards, notifications, Board Pack sections, agent commentary), their voice should be distinguishable, not interchangeable. The personas defined in the product spec apply to writing as well as voice.

**George (the steady, considered baseline):**
- Sentence length: medium, well-formed
- Vocabulary: simple, precise, occasionally a slightly old-fashioned turn of phrase
- Personality: warm, occasional dry humour, never first-person plural unless inclusive
- Example: "Worth keeping an eye on this one. Supplier concentration tends to creep up before it becomes a problem."

**Margot (the strategic, forward-leaning):**
- Sentence length: tends shorter, more direct
- Vocabulary: sharper, more active verbs, business-fluent without jargon
- Personality: asks questions, frames trade-offs, slightly impatient with vagueness
- Example: "Three objectives sounds like four. What would have to be true for the housing pipeline to pay back inside twelve months?"

**Iain (the cautious, considered):**
- Sentence length: longer, more qualified
- Vocabulary: precise, occasionally formal, careful with absolutes
- Personality: measured, lays out what's known and unknown, weighs likelihoods
- Example: "On balance, the Acme exposure looks more material now than it did in March. Not urgent, but worth a conversation in the next fortnight."

**Priya (the pragmatic, modern):**
- Sentence length: varied, conversational
- Vocabulary: clear, modern, technology-aware but never tech-bro
- Personality: cuts through complexity, names the actual problem, suggests practical next steps
- Example: "Your team's still emailing snag lists around. That's the bottleneck, not the AI question. Fix that first."

**System messages (notifications, errors, confirmations):**
- No agent voice — neutral and direct
- Still warm but unattributed
- Example: "We've saved that. You can find it in your risk register."

### 9.5 Email-specific tone

Emails have less context than in-app messages — the leader may not be in Oracle mindset when they open one. Account for this:

- Always attributable to a named agent (usually George, occasionally a specialist)
- Subject lines: descriptive, not clever
  - Good: "George — your May read is ready"
  - Bad: "Oracle has news for you!"
- First line restates context briefly ("From your fractional advisor George at Oracle.")
- Single CTA, no link soup
- Sign-off: "— George" (em-dash, name, no honorifics)

### 9.6 Things never to write

- "AI-powered" anywhere in user-facing copy
- "Onboarding" as a noun ("complete your onboarding")
- "Insights" (a vague consultant cliché)
- "Optimise" / "Leverage" / "Synergy" / "Stakeholder" / "Best-in-class"
- "Hi there!" / "Welcome back!" / any default greeting with no specificity
- Exclamation marks (one might be acceptable per month of product use)
- Emoji in the product itself (acceptable in marketing, never in app copy)
- "Powered by..."

### 9.7 Names

- The product is **Oracle** (placeholder; may change)
- The Advisor is **George**, never "your AI assistant", never "the chatbot"
- The specialists are **Margot, Iain, and Priya**, named when invoked, never grouped as "the AIs"
- Anthropic, ElevenLabs, and other vendors are invisible in user-facing copy

## 10. Iconography and imagery

### 10.1 Icons

- Library: **Lucide React**, exclusively.
- Style: outline only (never filled).
- Weight: 1.5px stroke.
- Size: 16px (inline), 20px (UI controls), 24px (large affordances).
- Colour: inherits text colour. Never coloured for decoration.

**When icons are used:**
- Navigation (settings, profile)
- Toolbar actions inside expanded views
- The handoff card (a small arrow indicating "outward to Valorem First")
- The walkthrough play indicator (a thin triangle)
- Error states (wifi-off, etc.)

**When icons are not used:**
- As decoration in section headers
- On buttons (text only)
- In the card of the moment
- In the Board Pack
- On the home canvas

### 10.2 Imagery

There is no imagery in the product. No stock photos, no illustrations, no marketing visuals.

The only visual elements are:
- The orb
- Charts (treated as editorial exhibits)
- Typography
- The thin teal accents

This is intentional. The product feels premium because of the restraint.

### 10.3 Branding

- The Oracle wordmark: serif, all lowercase, optical kerning, in `--color-ink`. Final mark to be designed.
- Never paired with a symbol/icon mark. The orb is the symbol, but it's not a logo.
- The wordmark appears in the top nav and at the bottom of the Board Pack. Nowhere else.

### 10.4 Favicon, OG image, system meta

- Favicon: a simplified circular gradient representing the orb. Two sizes (32px, 16px). Single colour fallback for older browsers (solid teal circle).
- Apple touch icon: 180x180, the orb on `--color-bg` background, generous padding (orb takes 60% of icon area)
- Open Graph image: 1200x630, the orb prominent on the left, the Oracle wordmark and a single line of text on the right ("Your fractional business advisor")
- Theme colour (system / browser UI): `--color-bg` for light mode (no dark mode support in PoC)

### 10.5 Print styles

Some leaders will print Board Packs. Without print styles, the dark teal and shadows render poorly. Required:

```css
@media print {
  /* Force light theme */
  body { background: white; color: black; }

  /* Remove shadows and decorative elements */
  * { box-shadow: none !important; }

  /* The orb does not print */
  .orb, .orb-container { display: none; }

  /* Walkthrough chrome (progress bar, indicators) does not print */
  .walkthrough-chrome { display: none; }

  /* Section dividers print as simple horizontal rules */
  .section-divider { border-top: 1px solid #999; }

  /* Charts: print as outline only, no fills */
  .chart-fill { display: none; }
  .chart-stroke { stroke: black; }

  /* Body text: ensure readable size */
  body { font-size: 11pt; line-height: 1.5; }

  /* Page breaks: avoid breaking inside sections */
  .board-pack-section { page-break-inside: avoid; }
}
```

Print styles apply only to the Board Pack route. All other pages render a "Print not available" message if attempted.

## 11. Accessibility

A premium product is accessible by default. Not as an afterthought.

### 11.1 Colour contrast

All text-to-background combinations meet WCAG AA minimum:
- `--color-ink` on `--color-bg`: 14.3:1 ✓
- `--color-ink-2` on `--color-bg`: 8.1:1 ✓
- `--color-ink-3` on `--color-bg`: 4.9:1 ✓
- `--color-teal` on `--color-bg`: 8.7:1 ✓
- White on `--color-teal`: 8.7:1 ✓

`--color-ink-muted` is intentionally below AA and is only used for disabled states, never for content.

### 11.2 Focus states

Every interactive element has a visible focus state:

```css
:focus-visible {
  outline: 2px solid var(--color-teal);
  outline-offset: 3px;
  border-radius: var(--radius-s);
}
```

Never `outline: none` without a replacement. Focus rings are designed, not removed.

### 11.3 Reduced motion

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

The orb stops breathing. State changes become instant colour shifts. Walkthroughs do not auto-scroll. The product remains fully functional and arguably feels even calmer.

### 11.4 Voice alternatives

Every voice-driven interaction has a text alternative:

- Onboarding can be completed by typing instead of speaking (text input replaces voice)
- The Board Pack walkthrough can be read silently
- Any agent message can be displayed in the transcript

The product is voice-first, not voice-only.

### 11.5 Keyboard navigation

- All actions reachable by Tab
- The orb is focusable; pressing Space starts a session
- Modals trap focus correctly
- Esc closes any overlay
- Cmd+K opens a command palette for power users (deferred to v1)

These four milestones are a working visual prototype of Oracle. They prove the design language works before the heavier engineering (voice, memory, post-processing) is invested in.

## 13. The visual-quality checklist

Before any screen ships, it must pass these checks:

- [ ] Could this be a screenshot from a Stripe / Linear / NYT app marketing page?
- [ ] Have I used only colours from the token palette?
- [ ] Is the serif used for content and the sans used for chrome?
- [ ] Is there enough whitespace? (If unsure: add more)
- [ ] Are there any borders that should be shadows?
- [ ] Are there any default Tailwind shadows? (Replace with tokens)
- [ ] Are any icons used decoratively? (Remove them)
- [ ] Does the copy sound like George wrote it, not a SaaS company?
- [ ] Does anything bounce, spring, or snap? (Remove or replace)
- [ ] If I removed three elements, would it look better? (Often yes)
- [ ] Does the screen pass the 9pm-with-a-glass-of-wine test?

If any answer is no, redesign before moving on.
